What this dashboard shows (in plain language)

  1. Big-O describes how runtime grows when the input size n gets larger — for example, doubling n might multiply time by 4 for an O(n²) algorithm.
  2. Why pin a physical core? Normally the operating system moves your program between cores. Each move can clear CPU cache and add random jitter. This benchmark dedicates one physical core so timings reflect the algorithm, not core hopping.
  3. What we measure is wall-clock time in milliseconds, repeated several times; we keep the median so one noisy run does not skew the curve.
